Low‑Cost Persistent Storage and Hosting for Small Vue/Node/TypeScript Apps (NZ – 2025)

Context and constraints

You maintain several single‑page applications (SPAs) built with Vue and Node/Express/TypeScript. At any time you can only afford to operate two apps concurrently and you need to switch other apps in and out of live mode without incurring additional charges. Persistent storage is provided by SQLite in your local environment, but you need a reliable, low‑cost production solution (≈ NZ$8–10 or <US$5 per app/month) with flexibility to host only two live apps at a time. The recommendations below are based on September 2025 information gathered from current provider documentation and comparative articles.

Understanding SQLite in production

SQLite is an embedded database that stores data in a single file and is included as a library in your application. It is widely used because it is lightweight and has a good SQL feature set, but it was designed for single‑user applications. Running SQLite in a server environment works well when concurrency is low and you can control write access; however you must protect against data loss. Fly.io’s engineering blog notes that developers increasingly place SQLite at the centre of the stack and use Litestream to continuously replicate the write‑ahead log to an object‑storage service like Amazon S3, making replication “remarkably cheap”[1]. This approach provides a fully self‑hosted database with automatic backups, but it doesn’t solve multi‑user concurrency or scaling issues.

When SQLite makes sense

·         Low concurrency – You control the code and can serialise writes.

·         Small datasets – Datasets are under a few gigabytes and fit on a low‑cost VPS disk.

·         Cheap backups – Replication to S3 or Backblaze B2 costs cents per month[1].

·         Edge or serverless – New platforms like Cloudflare D1 (serverless SQLite) allow scaling across global regions.

When to use a managed relational database

If you need multiple concurrent connections (for example, multiple users writing data simultaneously), or you want built‑in authentication, replication and scaling, consider a managed PostgreSQL/MySQL database. The “Tiny Turtle” free plan on ElephantSQL provides a 20 MB database with 5 concurrent connections[2], while paid plans start at US$5/month (Simple Spider) for 500 MB and 10 concurrent connections[3]. Neon (serverless Postgres) has a free tier with 0.5 GB of storage per project, 100 compute‑unit hours and scale‑to‑zero after five minutes of inactivity[4]; it automatically scales compute up to 2 vCPU/8 GB RAM and supports branching for staging/dev. Supabase’s free plan offers a Postgres database with up to 500 MB of data, unlimited API requests, authentication and storage (1 GB file storage and 2 GB transfer)[5], while ElephantSQL and DigitalOcean provide paid tiers that remain within your budget (~US$5 per month).

Low‑cost hosting platforms

Hosting cost is often higher than database cost. You need to run only two live apps at a time, and be able to deactivate others. The following options meet the budget or have generous free tiers:

Fly.io

·         Free tier: 3 “shared‑CPU‑1x” instances with 256 MB RAM. Additional vCPUs or memory are paid per second.

·         Supports Docker deployment and private networking; integrated volumes for persistent storage. Fly’s blog demonstrates replicating SQLite with Litestream, replicating to S3 cheaply[1].

·         Suitable for running a Node/Express API and SQLite database in one container. You can scale to zero manually by stopping the app when not in use (no cost when stopped).

Railway & Render

·         Railway’s free tier (2025) offers 500 hours of compute per month with a sleep‑after‑30‑minutes policy—good for prototypes but may sleep if idle; paid plan is about US$5/month for 1 GB RAM, which fits your budget.

·         Render’s free tier for web services provides 750 hours/month with automatic sleep after 15 minutes of inactivity; the Starter plan costs US$7/month for 512 MB RAM, free team collaboration and continuous deployment. Both support Postgres add‑ons and environment variables to scale to zero.

DigitalOcean droplet (shared VM)

·         Small droplets start at US$5/month (1 vCPU, 1 GB RAM, 25 GB SSD). You can run multiple apps (each in a Docker container) on one droplet, controlling which containers are active. Combined with ElephantSQL or SQLite + Litestream, this meets your budget for two apps.

Cloudflare Pages + Workers + D1

·         Free tier includes unlimited static sites and 100 000 worker requests/day. D1 (SQLite) provides 1 GB storage and 25 million read/write operations per month free. Good for SPAs and serverless APIs; switching off an app is as simple as unbinding its Worker.

Netlify/Vercel

·         Free tiers allow deployment of SPAs with continuous integration and preview environments. Edge functions (serverless) have limited free executions per month; paid plans start around US$20/month—above your budget. Use these for the front‑end only and host the backend elsewhere.

Designing a cost‑conscious architecture

1. Consolidate hosting with containerisation

Use a single $5–7/month VPS (e.g., DigitalOcean droplet or Lightsail) running Docker and Nginx. Package each application (Vue static front‑end + Node/Express API) into its own Docker image. Use docker‑compose to define services and volumes. Configure Nginx to route domains to the appropriate container.

·         Switching apps live – Start or stop containers using a simple script or the Docker API. When an app is inactive, its container can be stopped to free memory/CPU. Its SQLite database remains on disk; ensure periodic backup via Litestream to S3/B2 for a few cents per month. Starting an app takes seconds.

·         Database options – For apps with low write concurrency or small datasets (e.g., personal projects), continue using SQLite with Litestream replication. For apps requiring more connections or remote access, use an external Postgres service like ElephantSQL or Neon.

2. Serverless relational options

For an app with light traffic but requiring multiple concurrent connections, use Neon’s free plan. Neon scales compute to zero after five minutes and charges only when the DB is used[4]. Since your apps may be inactive most of the time, this effectively offers a free persistent Postgres database. When your app becomes active, Neon automatically spins up compute. Combine Neon with Cloudflare Pages/Workers or Fly to host the front‑end/backend functions. Supabase’s free tier is similar but includes authentication, storage and real‑time features[5]; however compute always runs, so ensure your usage stays within the free quotas.

3. NoSQL for non‑relational workloads

If some apps simply store JSON or key–value data (e.g., caching or user preferences), consider AWS DynamoDB. The perpetual free tier provides 25 GB storage and 25 WCU/RCU per month[10]. Combined with API Gateway/Lambda, you can host a fully serverless backend with no server cost. DynamoDB’s on‑demand mode charges per request ($0.25 per million read/write units)[11], so minor traffic incurs pennies.

Switching between live apps

1.      DNS or reverse‑proxy switch – Point your live domain at the container you wish to activate. Use Nginx or Traefik with per‑app configurations. Inactive apps remain stopped to save CPU/memory.

2.      Database snapshotting – Before stopping an app, ensure the SQLite DB is flushed and replicate the latest WAL to S3/B2 using Litestream. For Postgres (ElephantSQL/Neon), create a snapshot or rely on automated backups.

3.      Environment variables – Parameterise your Node app to connect to the correct database (SQLite file or external Postgres). When switching, update the .env file and restart the container.

4.      Automation – Write scripts to stop one container, back up the DB, update DNS records, and start another container. Cron jobs can automatically stop idle apps after certain hours.

Recommendation summary

·         For very small or experimental apps: continue using SQLite. Deploy the app on Fly.io (free for small instances) or a $5 VPS. Use Litestream to replicate the SQLite file to S3 or Backblaze B2; replication cost is extremely low and provides durability[1].

·         For relational apps requiring concurrency: move to a free/low‑cost managed Postgres. Neon’s free tier provides 0.5 GB storage with scale‑to‑zero and is perfect for intermittent apps[4]. Supabase offers a richer free tier (500 MB DB, auth, storage)[5] but may cost more if you exceed quotas. ElephantSQL’s Tiny Turtle is too small for serious use but its Simple Spider plan at US$5/month (500 MB, 10 connections)[3] fits within your budget for an app that must always be on. For a bigger app, you can upgrade to Crazy Cat (1 GB) for US$10/month[3].

·         For non‑relational or caching workloads: use AWS DynamoDB. The free tier offers 25 GB storage and 25 WCU/RCU per month[10] and scales to high throughput. This is ideal for key‑value data or event logging.

·         Hosting choice: run your apps in Docker containers on a small VPS or on Fly.io. Use Nginx as a reverse proxy to route requests. For static front‑end assets, use Cloudflare Pages or Netlify’s free tier. Only start the containers for the two live apps; stop others to avoid resource usage.

By combining low‑cost hosting with free database tiers, careful switching, and SQLite replication, you can run two production apps concurrently for approximately US$5 per app per month while retaining the ability to swap other apps in and out of production with minimal overhead.
